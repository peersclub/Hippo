The model — a parasite on a real host

Assetworks Exchange is a self-contained venue with its own signed trade API, fill engine, balances and positions, streaming live prices from Binance public feeds. Hippo embeds as a parasite via one <script> tag and trades through the execution seam. Because the host speaks a standard HMAC-signed venue wire, the parasite drives it through production code paths — HMAC signing, capability discovery, the poll reconciler, both confirm surfaces — not a sim timer.

The payoff: an order the conversational parasite places lands in the same book, same wallet (trader-1) as the host's own order ticket, so it shows in the native blotter and moves the same balances — exactly like a real third-party integration.

Two order paths, one book

PathHowAuth
HUMANThe order ticket on the exchange UI (spot/perp, buy/sell, market/limit)First-party /ui/orders — unsigned
PARASITETalk to Hippo: “buy 0.01 BTC” → ticket → confirmHMAC-signed /api/v1/trade/orders via the seam

Both resolve to the same trader-1 wallet, so anything placed either way appears in the host Open orders / Positions / Balances blotter (live over SSE) and in Hippo's positions card.

Non-obvious, by design: a market order is not instant — it rests ~2.5–3s so the parasite's reconciler (polls every 2s) can observe it fill. Liquidation price is display-only — the engine never force-liquidates. The balance figure includes reserved funds, so it won't visibly drop when a resting/perp order locks margin.

Confirm surfaces — the admin switch Open Decision #6

Open the gear → venue admin drawer on the :4001 host and flip Confirm surface (it writes host-venue /admin/config; the parasite reads it live at confirm time — no restart). In local dev this endpoint is unauthenticated.

SurfaceBehaviour
api (default)Hippo confirms in its own panel and places directly with the scoped key. The order then appears in the host blotter.
js_callbackHippo hands off; the host renders its own confirm modal. Nothing places until you approve there. Decline → parasite shows “DECLINED ON THE VENUE”; timeout (90s) → “CONFIRMATION TIMED OUT”.

Conversational AI — talk to Hippo on :4001 or :4000

  1. Ask why is BTC moving today? a skeleton, then prose streams token-by-token; the model tag anthropic/claude-haiku-4.5 shows top-right of the card during streaming and on the final card. Real price / 12h / funding stats + sparkline, all matching the market-data feed
  2. Ask a concept question: what is a funding rate? a CONCEPT card — prose only, empty stats grid, source chip “HIPPO KNOWLEDGE”. (Numbers are always retrieval from the feed; the model only writes prose, so it can't fabricate a price.)
  3. Ask should I buy BTC now? an advice-decline card — “◇ NO ADVICE — BY DESIGN” — pivoting to live facts. Enforced on the model's output, not just the prompt
  4. Ask the same market question twice within 5 minutes. the 2nd returns instantly with a “CACHED BRIEF · updated just now” badge (TTL scales with volatility: 45s hot → 300s calm)
  5. On a finished brief, tap ↻ REFRESH. the button holds “↻ REFRESHING…”, then the same card updates in place (a new brief carrying replaces:<id>) — it does not stack a new card below
Known limitation — language: the intelligence service can generate briefs in Hindi/Hinglish, but the gateway currently forwards the trader's language only to intent classification, not to the answer path — so through the SDK, brief and decline content comes back in English regardless of the selected language. Arabic (ar) is chrome + right-to-left layout only; it is not a brief-generation language. Multi-language answer content is not wired end-to-end yet.

Resilience & edge cases

Do thisExpected
Kill the AI service: pkill -f uvicorn, then ask a research questionNever-500: you still get a truthful market-data-only brief + an amber “HIGH MARKET LOAD” banner (once per episode). Orders and prices keep working. Restart with pnpm dev.
Restart the gateway while the panel is openThe SSE stream drops; the SDK re-mints a fresh session (backoff 1s→30s) instead of hanging on “Reconnecting”. On the in-memory dev stack thread history is lost on that restart (durable cold-resume needs Redis).
Leave an order in “WAITING FOR YOUR CONFIRM” with no venue eventAfter TICKET_EVENT_TIMEOUT_MS (10 min) a backstop closes the card with a terminal “expired” frame — it can't hang forever.
Stop the server mid-stream20s after the last delta a client watchdog finalizes the card (cursor stops) instead of blinking forever.
Queue actions offline (chip tap / feedback / settings), then reconnectQueueable uplinks buffer (FIFO, cap 20) with an “{n} QUEUED” row and drain in order on reconnect. Trading actions are never queued — they fail loud.
Prod-only behaviours: the 401 “blocked” quiet-disable and 429 “busy this month” capacity notice fire against the production gateway (HIPPO_DEV=0) — on the local dev stack a bad key silently falls back to the demo partner, so you won't see “blocked” here. Redis is optional: unset today (in-memory sessions/cache); add REDIS_URL only for multi-instance or restart-durable deploys.

Partner portal — localhost:5176

The partner-facing plane, a separate trust boundary from the operator admin. There is no public sign-up: an operator mints a one-time invite on a partner's admin detail page, the partner claims it at #/claim (token min 16 chars, 12-char-min password), then signs in.

PageWhat a partner sees
OverviewTheir own venue name + status, MAU this month vs quota, known users, plan. Sandbox partners see a “production approval pending” note.
IntegrationTheir embed key (pk_…) + a copy-able <script> tag; admin seats can rotate the secret (shown once, audited without the value).
PlanAssigned plan + entitlements; admin seats get “Request a change”.
ActivityAudit rows scoped to this partner.
Tenancy: no portal route accepts a partner id — it comes only from the signed session cookie, so cross-tenant reads are unexpressible. Viewer seats are read-only (mutations 403); only admin seats can change anything. A suspended partner's admins can't sign in at all (403).
